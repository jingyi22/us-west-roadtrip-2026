## 🔍 自我复盘：做得好的地方

### 1. 主动确认后再执行
在移动文件前，先列出了清晰的整理框架发给用户确认，而不是闷头执行。这避免了返工。

### 2. 文件名语义化
使用编号前缀（01, 02, 03...）保证文件夹在系统排序时按项目顺序排列，一目了然。

### 3. README 结构完整
每个 README 包含了：项目概述、主要文件清单、核心功能/发现，便于日后追溯。

### 4. 遗留文件标注清楚
主动告知用户还有 `pptx_extracted/`、`read_stdin.py`、`WPS同步文件夹.pdf` 等不在整理范围内的文件，让用户决定如何处理。

## 🔍 自我复盘：需要改进的地方

### 1. 整理任务的触发时机
文件散落的问题其实从4月18-20日就已经存在了，但我没有主动提出整理，而是在用户明确要求时才做。以后在完成一个项目后，应该及时整理，而不是等积累多了再处理。

### 2. 缺少统一的文件命名规范
workspace 根目录存在多个版本的文件（如 `TripGenie_AI_Prototype.html` vs `TripGenie_v3.html` vs `TripGenie_fixed.html`），我自己也有点分不清哪个是最新/最稳定的版本。以后在项目内应该明确标注版本状态（如 `*_latest*`、`*_broken*`）。

### 3. 大文件未及时转移
`AI_Planner_Competitor_Research_20250522.pptx` (75MB) 一直在根目录，这个 PPTX 本来是从飞书下载的，可以考虑是否需要保留，还是上传到飞书云端后删除本地副本。

### 4. 备份文件过多
项目内存在较多 backup/old 版本，虽然保留了历史记录，但长期来看会增加查找成本。可以考虑用 git tag 或 dated backup 方式管理。
